Page → shell forwards `isOwner` (`project.ownerId === identity.userId`); the navbar `Share` button opens `<ShareDialog>` which fetches `/api/projects/[projectId]/collaborators` on open. Owners see the invite form and per-row remove buttons (optimistic with revert); collaborators see the list only and no actions. Collaborator rows render Clerk display name + avatar when available and fall back to the raw email — there is intentionally no local users table; `lib/collaborators.ts` resolves names/images through Clerk's backend `users.getUserList({ emailAddress })` at request time, and every lookup or Clerk-client init failure degrades to `{ name: null, imageUrl: null }`. The auth route reuses `getProjectForUser` from spec 08 so owner/collaborator gating stays in one place — `defaultAccesses: []` keeps every room private and the auth endpoint is the sole gate. Spec 10 mentioned "All required Liveblocks packages are already installed" but `@liveblocks/node` was missing and had to be added.
- Canvas wrapper is two files by design: `canvas-room.tsx` owns the providers/Suspense/ErrorBoundary chrome and the layout (rounded card under the navbar), `canvas-flow.tsx` owns the React Flow + `useLiveblocksFlow` binding. Splitting keeps the suspending child below the boundary and lets future canvas chrome (toolbar, presence) attach without touching the flow itself. `useLiveblocksFlow` is invoked with `suspense: true` and empty `nodes`/`edges` initials so multiple clients joining an empty room don't race on a non-empty bootstrap. Imports the three required CSS bundles (`@xyflow/react`, `@liveblocks/react-ui`, `@liveblocks/react-flow`) from `canvas-flow.tsx` only. `Presence` already includes `isThinking: false` from spec 10's `Liveblocks` types, so `initialPresence` sets both `cursor: null` and `isThinking: false` to satisfy the strict typing. `ConnectionMode.Loose` (per spec) lets handles connect freely from any side. Imports for `LiveblocksProvider`, `RoomProvider`, and `ClientSideSuspense` come from `@liveblocks/react/suspense` — using the suspense export is required for the suspending Liveblocks Storage / React Flow hooks downstream. `react-error-boundary` was added as a new dependency since the Liveblocks docs prescribe it for connection-failure fallbacks. The previous `CanvasSurface` placeholder is deleted, not preserved — the new `CanvasRoom` replaces it inline in `EditorWorkspaceShell`. `roomId` is the existing `activeProjectId` (room ID == project ID, per spec 07).
- Auth flow switched from ID tokens to access tokens. Spec 10 originally called `liveblocks.identifyUser(userId, ...)` with `getOrCreateRoom(roomId, { defaultAccesses: [] })`. That combination produced WebSocket close code 4001 ("You have no access to this room") because the room is private and the user is never added to `usersAccesses`. Rather than maintaining `usersAccesses` per collaborator on every auth call, the route now uses `liveblocks.prepareSession(userId, { userInfo }).allow(roomId, session.FULL_ACCESS)` and returns the result of `session.authorize()`. The room ACL is gone — `getProjectForUser` in our route is the single gate, computed per request from Prisma + Clerk. Rooms now auto-create on first connection (no `getOrCreateRoom` call). The `userInfo` shape (`{ name, avatar, color }`) is identical between the two flows, so client-side `useSelf`/`useOthers` consumers are unaffected.
- Shape panel (spec 12): bottom-center pill toolbar lives inside the canvas surface, rendered as a sibling of `<ReactFlow>` (not as a React Flow `Panel`), so its DOM is outside the flow viewport but visually overlays it via absolute positioning. `canvas-flow.tsx` is now wrapped in `<ReactFlowProvider>` and split into `CanvasFlow` (provider) + `CanvasFlowInner` (`useReactFlow` + `useLiveblocksFlow`); this is required because `useReactFlow().screenToFlowPosition` must be called from inside the provider context that also owns the drop wrapper. Drag payload uses a custom MIME type `application/x-canvas-shape` (`CANVAS_DRAG_MIME_TYPE`) to avoid clashing with React Flow's built-in DnD events, and carries `{ shape, width, height }` (`CanvasShapeDragPayload`). Default sizes live in `NODE_SHAPE_DEFAULT_SIZES` in `types/canvas.ts` (rectangles wider than tall, circles square, diamonds slightly larger). Node IDs are `${shape}-${Date.now()}-${counter}` where the counter is a module-scoped integer — the timestamp keeps IDs unique across reloads, the counter keeps multiple drops in the same millisecond unique. New nodes are added via `onNodesChange([{ type: "add", item }])` rather than a `setNodes` setter; this routes the add through the Liveblocks-bound change handler so it persists to Storage. The custom `CanvasNode` renderer is registered in a module-level `nodeTypes` map (stable reference) and currently renders every shape as a bordered rectangle with the label centered — shape-specific SVGs (diamond, hexagon, cylinder per `ui-context.md`) are intentionally deferred. Lucide's `Square` icon is used for the rectangle since Lucide has no `Rectangle` icon. Drop position centers the node on the cursor by subtracting half the width/height. Drop payload is validated (shape ∈ `NODE_SHAPES`, width/height are numbers) even though it originates from our own UI — JSON in `dataTransfer` is treated as untrusted input per code-standards.
