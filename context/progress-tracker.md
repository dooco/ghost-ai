## Session Notes

- Editor page now includes full base workspace frame: left floating project sidebar, center canvas scaffold, and right slide-over AI panel shell.
- Auth wired: Clerk dark theme from `@clerk/ui/themes`, CSS variable overrides, `proxy.ts` route guard, `/sign-in` and `/sign-up` two-panel pages, `/` redirect logic, `UserButton` in navbar.
- Editor home is now the center content of `/editor`; the previous `CanvasSurface` scaffold remains in the codebase for later use once a project-open state exists. Project sidebar consumes mock data from `lib/mock-projects.ts`; owned vs shared is the sole gating signal for showing rename/delete actions. Dialog state is centralized in `useProjectDialogs` and lives at `EditorWorkspaceShell`.
- Prisma data layer baseline is now in root runtime paths, not context-only paths: schema/model files are under `prisma/`, client singleton is in `lib/prisma.ts`, and the initial migration has been applied with generated client output in `generated/prisma`.
- Project API routes are backend-only and not wired into the editor UI — the sidebar still consumes mock data from `lib/mock-projects.ts`. Middleware (`proxy.ts`) now skips `auth.protect()` for `/api/*` so route handlers control the response code for unauthenticated/forbidden cases; Clerk continues to protect every non-API route except sign-in/sign-up.
- Editor home (`/editor`) and per-project workspace (`/editor/[projectId]`) are server components that fetch through `lib/projects.ts`; mock data has been removed. Project ID is now intentionally the Liveblocks room ID (`${slug}-${suffix}`), generated client-side in `useProjectActions` and persisted via the optional `id` field on `POST /api/projects`. `Project` schema is unchanged — the slug/room ID lives inside the existing `id` column rather than a separate field. `useProjectActions` is the only dialog hook; create routes the user to `/editor/${projectId}`, rename refreshes, and delete redirects to `/editor` only when the deleted project equals `activeProjectId` (otherwise refreshes). The `[projectId]` workspace page currently checks ownership only and renders the same shell as the home — a real workspace surface (canvas load, collaborator access) is still to come.
- Workspace route is now `/editor/[roomId]` (spec 08 aligned the segment name with the Liveblocks room — the underlying value is still the project ID). Access is centralized in `lib/project-access.ts`: pages call `getCurrentIdentity()` first, redirect to `/sign-in` on null, then call `getProjectForUser(roomId, identity)` and render `<AccessDenied />` (link back to `/editor`) on null. Owner-only restriction is gone — collaborators (matched by primary email against `ProjectCollaborator.email`) can now load the workspace. The shell now branches on `activeProjectId`: with a project, it renders `CanvasSurface` in the center and mounts the right `AiSidebarShell` whose visibility is driven by a new `isAiSidebarOpen` state toggled from the navbar's `Sparkles` button; without one, it keeps the `EditorHome` center and hides the AI panel entirely. The navbar's middle column shows the project name; right-side `Share` and AI-toggle buttons only render when a project is active. Sidebar items receive `activeProjectId` and gain an `aria-current="page"` highlight (subtle ring + elevated background). Sharing behavior and real canvas/AI logic remain unimplemented per spec 08 scope.
- Sharing is now wired end-to-end. Page → shell forwards `isOwner` (`project.ownerId === identity.userId`); the navbar `Share` button opens `<ShareDialog>` which fetches `/api/projects/[projectId]/collaborators` on open. Owners see the invite form and per-row remove buttons (optimistic with revert); collaborators see the list only and no actions. Collaborator rows render Clerk display name + avatar when available and fall back to the raw email — there is intentionally no local users table; `lib/collaborators.ts` resolves names/images through Clerk's backend `users.getUserList({ emailAddress })` at request time, and every lookup or Clerk-client init failure degrades to `{ name: null, imageUrl: null }`. Emails are stored lowercase and validated server-side; the API returns `409` for duplicate invites and `403`/`404` paths match the existing project routes' shape. The `DELETE` endpoint uses `?email=` rather than a path segment to avoid issues with dot characters in dynamic segments. Project URL copy uses `navigator.clipboard.writeText` with a 1.5s `Copied!` toast — silently no-ops in non-secure contexts where clipboard is unavailable.
